# KBO 올타임 드래프트 게임 — 기획/설계 문서

> 가칭: **144-0** (한 시즌 무패 우승 = 정규시즌 전승)
> 원본 레퍼런스: [38-0.app](https://38-0.app) (English top-flight 드래프트 게임)
> 작성일: 2026-06-13

---

## 1. 한 줄 요약

> KBO 역대 모든 시즌(1982~현재)의 선수 데이터로 **올타임 드림 로스터를 드래프트**하고,
> 정규시즌 144경기를 시뮬레이션해 **몇 승까지 가능한지(전승 가능한지)** 도전하는 무료 웹 게임.

---

## 2. 게임 컨셉

### 2.1 핵심 루프
```
[휠 돌리기] → 구단 + 시즌 당첨 (예: 2017 KIA, 1992 롯데)
     ↓
[드래프트] 해당 시즌 로스터에서 선수 1명 선택
     ↓
[슬롯 배치] 내 로스터의 빈 포지션 슬롯에 배치
     ↓
(로스터 완성될 때까지 반복)
     ↓
[시즌 시뮬레이션] 144경기 진행 → 성적/순위/승수 확인
     ↓
[결과 공유] 전승 도전 결과를 카드 이미지로 공유
```

### 2.2 38-0과의 차이 (야구 특성 반영)
| 항목 | 38-0 (축구) | 144-0 (야구) |
|------|-------------|--------------|
| 로스터 크기 | XI (11명) | **풀 로스터 (26~28명 수준)** |
| 포지션 | GK/DF/MF/FW | 타자 8 + 지명타자 + 선발 5 + 불펜/마무리 |
| 경기 수 | 38경기 | **144경기** (시즌별 가변: 과거 126·128·133) |
| 시뮬 모델 | 팀 강도 기반 | **타순/이닝 기반 세이버메트릭 시뮬** |
| 데이터 시작 | 1992-93 | **1982** (KBO 원년) |

### 2.3 풀 로스터 드래프트 구성 (확정)
선택된 플레이 단위. 최종 로스터 슬롯:

**야수 (Position Players) — 13명 내외**
- 포수 C ×1
- 내야수 1B / 2B / 3B / SS ×4
- 외야수 LF / CF / RF ×3
- 지명타자 DH ×1
- 백업(유틸) ×4

**투수 (Pitchers) — 13명 내외**
- 선발 로테이션 SP ×5
- 셋업/중간계투 RP ×6~7
- 마무리 CL ×1

> 슬롯 수는 밸런싱에 따라 조정. MVP에서는 **선발 라인업 9 + 선발 로테이션 5 + 마무리 1 = 15슬롯**으로 단순화 후 확장 권장.

## 3. 데이터 (가장 중요한 파트)

게임 로직보다 **선수 시즌별 스탯 데이터 확보가 프로젝트의 80%.**

### 3.1 데이터 소스 후보
| 소스 | 강점 | 비고 |
|------|------|------|
| **Statiz (스탯티즈)** | WAR/wRC+/FIP 등 세이버 지표 완비, 1982~현재 | 비공식, 크롤링 정책 확인 필요 |
| **KBO 공식 기록실** (koreabaseball.com) | 1차 출처, 신뢰도 높음 | 세이버 지표는 부족, 파싱 까다로움 |
| **MyKBO Stats** | 영문/정리된 시즌 데이터 | 커버리지 확인 필요 |
| **Baseball Reference (KBO)** | 일부 KBO 데이터 영문화 | 커버리지 제한적 |
| **Retrosheet 류 / 공개 데이터셋** | 가공 용이 | KBO는 제한적 |

> ⚠️ **법적/이용약관 확인 필수**: 크롤링 전 각 사이트 robots.txt 및 이용약관 검토. 비상업·팬 프로젝트라도 데이터 출처 명시 권장. 38-0도 "독립 팬 제작, 비제휴" 명시함.

### 3.2 필요한 스탯 (시뮬 입력값)
**타자**
- 기본: PA, AB, H, 2B, 3B, HR, BB, HBP, SO, SB, CS
- 파생: AVG, OBP, SLG, **wOBA**, wRC+, ISO
- 시뮬 핵심: 타석당 결과 확률 분포 (1B/2B/3B/HR/BB/HBP/SO/Out)

**투수**
- 기본: IP, H, HR, BB, HBP, SO, ER, BF
- 파생: ERA, **FIP**, WHIP, K%, BB%, K/9
- 시뮬 핵심: 상대 타자 결과 억제력

**보정 계수**
- 시즌별 리그 평균 (era adjustment) — 1982 타고투저/투고타저 보정
- 파크 팩터(선택) — 잠실 vs 사직 등

```jsonc

  "bat": {

    "HR": 26, "BB": 96, "HBP": 8, "SO": 91, "SB": 0,

  },
  "pit": null,
```

### 3.4 데이터 파이프라인
```
[수집 스크립트] (Node/Python) → 원본 raw HTML/JSON
     ↓ 정규화/검증
[정제 스크립트] → 표준 스키마 JSON
     ↓ 빌드타임 번들
[정적 데이터셋] (앱 내 /data/*.json 또는 분할 청크)
```
- 빌드타임에 정적 번들 → **런타임 서버 0, 비용 거의 0**
- 선수 수가 많으면(수천~만) 구단·시즌별 청크로 분할 lazy-load

## 4. 시뮬레이션 모델

야구는 축구보다 **수학적 모델링이 검증되어 있어** 유리.

### 4.1 모델 후보 (단계적)
**Level 1 — Pythagorean (MVP 추천)**
- 팀 득점 RS / 실점 RA 추정 → 기대 승률 = RS^x / (RS^x + RA^x), x≈1.83
- 라인업 wOBA 합산 → 팀 득점력, 투수진 FIP → 실점력
- 빠르고 구현 단순, "느낌" 충분

**Level 2 — Markov Chain 이닝 시뮬 (목표)**
- 타순대로 타석 결과(1B/2B/3B/HR/BB/Out)를 확률 추출
- 베이스 상태(8가지) × 아웃카운트(3) 전이 행렬
- 9이닝 × 144경기 몬테카를로 → 사실적인 경기별 스코어
- 투수 교체 로직(선발 → 불펜 → 마무리) 반영

**Level 3 — 디테일 (선택)**
- 파크 팩터, 좌우 매치업, 수비(DRS/UZR), 부상/체력

### 4.2 상대 (리그) 설정
- 옵션 A: **가상 평균 리그** — 내 팀 vs 리그 평균 팀 144경기
- 옵션 B: **역대 명팀 9팀과 풀리그** — 더 재밌지만 데이터·밸런싱 부담
- MVP는 A 권장.

## 5. 기술 아키텍처

원본(38-0)이 **Next.js + 정적 데이터** 구조 (HTML에서 `_next`, turbopack 확인됨). 동일 방향 권장.

```
Frontend:  Next.js (App Router) + React + TypeScript + Tailwind
State:      Zustand 또는 React state (드래프트 진행 상태)
Sim:        순수 TS 모듈 (Web Worker에서 실행 → UI 블로킹 방지)
Data:       빌드타임 정적 JSON (구단/시즌별 청크)
Sharing:    OG 이미지 동적 생성 (결과 카드)
Hosting:    Vercel/Netlify (서버리스, 무료 티어)
PWA:        manifest + 오프라인 (원본도 PWA)
```

### 5.1 디렉터리 구조 (제안)
```
kbo-game/
├── DESIGN.md              ← 본 문서
├── app/                   # Next.js 라우트
├── components/            # 휠, 드래프트 보드, 라인업, 결과
├── lib/
│   ├── sim/               # 시뮬레이션 엔진 (Pythagorean → Markov)
│   ├── draft/             # 휠 로직, 가중치, 슬롯 검증
│   └── data/              # 데이터 로더/타입
├── data/                  # 정적 선수 JSON (빌드 번들)
├── scripts/               # 데이터 수집·정제 (별도 실행)
└── public/
```

### 9.1 진행 업데이트 (실데이터 + UI 완료)

- ✅ **실데이터 인제스트**: Kaggle netsong 데이터셋 → **17,278 선수-시즌(1982~2025)** 정규화. 검증 통과(2015 테임즈 wOBA .533, 선동열 FIP 1.12 등 역대 시즌 정확 재현).
- ✅ **웹앱 스캐폴딩 (Vite + React + TS)**: 휠→드래프트→144경기 시뮬→결과 전체 플레이 가능. `npm run dev`. 빌드 51KB(gzip).
  - `src/App.tsx` 게임 플로우, `src/ui/*` 컴포넌트, `src/data/load.ts` 데이터 로더/리그 조립
  - 상대 리그: 룰렛 연도 추첨 → 그 해 실제 10개 구단을 베스트 라인업으로 조립해 대결
- ✅ **검증**: 2017 KIA 실제 우승 라인업 자동 조립 + 시뮬 정상 동작

- ✅ **슬롯 수동 선택 UI**: 선수 클릭 시 가능 슬롯 1개면 즉시 배치, 여러 개면 슬롯 선택 모달. (`src/App.tsx` pending 상태 + `slotLabel`)
- ✅ **연도별 리그상수 시대보정**: `src/sim/leagueTable.ts`가 데이터에서 시즌별 lgwOBA/lgERA/rPA/fipConstant 산출. 선수는 자기 시즌 리그 대비(wRC+/ERA+ 방식)로 평가 후 상대 연도 환경으로 환산. 검증: 2014 lgwOBA .351(타고투저)·2022 .315(저득점)·1982 .321 등 실제 역사 재현.

- ✅ **설정 화면 (38-0 스타일)**: `src/ui/Settings.tsx` + `src/settings.ts`
  - 로스터 구성(Simple/Full) · 난이도(이지3/노멀1/하드0 리롤, 하드는 스탯숨김) · 스탯표시 ON/OFF(블라인드) · 드래프트 모드(스쿼드우선/포지션우선) · 선수 스탯 기준(해당시즌/전성기) · 시대 프리셋+연도 슬라이더
  - 전성기 모드: `pid`로 통산 최고 WAR 시즌 환산(`buildPrimeIndex`/`resolvePrime`). 연도 범위로 휠·상대 리그 필터.
- ✅ **UI 가독성**: 선수명 색상 버그 수정, 타자=초록/투수=파랑 타입 색상 구분, 포지션 한국어(포수/유격수/선발/마무리…).

- ✅ **결과 공유 카드**: `src/ui/ShareCard.tsx` — Canvas로 전적+리그순위 PNG 렌더, 저장/공유(Web Share)/클립보드. 무의존성.
- ✅ **풀 리그 순위**: `simulateLeague` — 내 팀 + 상대 라운드로빈으로 전체 순위표 산출(`src/sim/season.ts`).
- ✅ **감독 모드**: `src/ui/SeasonPlay.tsx` + `src/tactics.ts` — 144경기를 상대별 시리즈로 진행, 시리즈마다 전략 토큰(에이스 집중/총력전/로테이션 휴식, 이월효과 有)을 한정 배분해 승부 개입.
- ✅ **상대 리그 선택**: 랜덤 시즌 / 연도 지정 (`opponentMode`).
- ✅ **라인업·로테이션 정렬**: `src/ui/Arrange.tsx` — 타순/선발순서 직접 지정. 시뮬이 **타순 PA 가중**·**로테이션 이닝 가중**으로 순서를 반영(`BAT_PA`/`ROT_W`).
- ✅ **중복 선택 차단**: `playerKey`(pid 기반)로 같은 선수 재드래프트 방지(+"선택됨" 표시).
- ✅ **휠 잠금**: 뽑을 선수가 있는데 안 고르면 메인 휠 잠금(무료 리롤 방지) — 재추첨은 리롤 소진으로만, 리롤 0이면 선수 선택 강제.

- ✅ **데이터 청크 분할**: 인제스트가 시즌별 청크(`public/data/seasons/<year>.json`) + 초소형 `index.json`(288B) + 사전계산 `league-table.json`(6.6KB) + `prime.json`(481KB, 전성기 모드 전용)을 생성. 전체 선수 데이터(`data/players.json`)는 node 스크립트용으로만 보관(웹 번들 제외).
  - **초기 로드 4MB → ~7KB** (index+테이블만). 선수 데이터는 선택한 연도 범위만 lazy-load, 상대 연도는 해당 청크만, 전성기 인덱스는 prime 모드일 때만.
  - 클라이언트 로더: `loadMeta`/`loadSeason`/`loadSeasonRange`/`loadPrimeIndex` (`src/data/load.ts`, 캐시 내장).

**남은 작업**: ① OG 메타이미지 ② 데이터 자동 갱신 파이프라인(아래 §10) ③ 파크 팩터/좌우 매치업(Level-3) ④ 불펜/마무리 순서 가중(현재 균등) ⑤ "전체" 범위 선택 시 44청크 일괄 로드 → decade 묶음 등 추가 최적화 여지.

## 10. 데이터 파이프라인 장기 전략 (Kaggle 의존 탈피)

Kaggle 데이터셋은 **남이 수동 갱신**하므로 신뢰할 수 없음 (2025시즌 끝나면 누가 올려줄지 불확실). 시즌마다 최신 데이터를 받으려면 자체 파이프라인 필요. 옵션:

| 방법 | 장점 | 단점 | 추천 |
|------|------|------|------|
| **Kaggle 계속** | 가장 쉬움, 과거치 완비 | 갱신 보장 안 됨, 최신 시즌 지연 | 과거(1982~) 백필 용도로만 |
| **KBO 공식 기록실 크롤링** | 1차 출처, 신뢰도 ↑ | JS 렌더링/구조 변경 리스크, 약관 | ⭐ 최신 시즌 갱신용 |
| **Statiz 크롤링** | 세이버 지표 완비 | 비공식, 차단/약관 리스크 | 보조 |
| **공개 API (있다면)** | 안정적 | KBO 공식 오픈API 제한적 | 조사 필요 |

**권장 아키텍처**: 과거 데이터는 Kaggle로 1회 백필(완료) → **매 시즌 종료 후 KBO 공식 기록실을 크롤링하는 증분 스크립트**를 cron(예: 매년 11월)으로 돌려 `players.json`에 append. 크롤링 시 **robots.txt·이용약관 준수, 요청 rate-limit, 출처 명시** 필수. 이 증분 크롤러는 `scripts/`에 별도 작성 예정.
